# Browser Tools — Shared Context

## Vocabulary

| Term | Definition |
|------|------------|
| **ghost process** | A Chrome instance with no open tabs but still holding the debugging port |
| **profile clone** | A copy of the Chrome user data dir used to carry cookies/sessions across runs |
| **network idle** | No network requests for >=500ms — used as page-ready heuristic |
| **scrape schema** | A JSON object mapping field names to CSS selectors and types |
| **action sequence** | A serializable array of typed action objects (click, fill, navigate, assert, wait) |
| **task spec** | A structured plan: intent, URL, schema/actions, success criteria, failure modes |
| **materialization** | Converting a TaskSpec into a concrete runnable action sequence |

## ADRs

### ADR-001: Resource Cleanup with try/finally
Every CDP session must use try/finally to close WebSocket connections. Leaked sockets accumulate and ghost Chrome processes prevent reconnection.

### ADR-002: Assert-First Schemas
Schemas carry their own success criteria. validateSchema() catches malformed schemas at call time; assertSchema() validates extracted data against requirements — required fields must be non-null, arrays must be non-empty where specified. Failed assertions throw pinpointed errors ("items[0].name: required but got null (selector .name matched nothing)") so the fix targets the selector, not the downstream pipeline.

### ADR-003: Actions Are Data
Publish workflows are arrays of typed objects, not scripts. Data is serializable; code is not. Action sequences can be stored in Supabase, sent over Realtime, generated by an LLM, and replayed deterministically.
